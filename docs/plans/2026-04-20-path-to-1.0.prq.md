# King's Road — Path to 1.0

**Created**: 2026-04-20
**Supersedes**: `2026-04-18-v1.0-beta-release.prq.md` (still valid for Phase 0–3 mechanics; this doc is the broader roadmap)
**Target**: `v1.0.0` — shipping on GitHub Pages (web) + signed Android APK + iOS TestFlight candidate
**Timeframe**: Autonomous execution; no fixed deadline

## What this document is

The prior 1.0-beta plan was exhaustive on *mechanics* (Zustand purge, SQLite migration, procgen kill) but missed many things that separate a "shippable beta" from a real 1.0:

- content volume and narrative craft
- playability polish (feel, pacing, onboarding)
- quality gates (perf budgets, crash telemetry, real-device validation)
- post-launch mechanisms (updates, content delivery, community loop)

This doc is the full macro/meso/micro decomposition. Tiers follow the repo's established quest taxonomy:

- **Macro** (1–2h authored equivalent; in dev-time, 1–5 days): cross-cutting initiative, requires branching design decisions, touches multiple subsystems
- **Meso** (15–45min authored; in dev-time, 4–12h): coherent feature or pass, must branch between "do it one way vs. another"
- **Micro** (5–15min authored; in dev-time, <4h): scoped fix or single-file change, linear execution acceptable

Phase numbers continue from the prior PRD so cross-reference stays intact.

## Priority legend

- **P0** — blocks 1.0 (gated). If missing, we do NOT ship.
- **P1** — ship-impacting but not blocking. Can slip to 1.0.1.
- **P2** — polish / nice-to-have. Opportunistic.
- **P3** — post-1.0 roadmap — track here to avoid forgetting, not for this release.

## Macro initiatives

Each macro is a full initiative with its own sub-plan doc that will be authored when the initiative starts.

### Macro M1 — Capacitor native persistence harden (P0)

**Why**: `.claude/state/task-batch/state.json`-style approaches don't survive native app-switching. The web `visibilitychange → flush` path (shipped #205) works on browsers; the native Capacitor pause/resume lifecycle needs the same treatment.

**Branches (decide at start)**:
- (a) Tie save flush to `@capacitor/app`'s `pause` event exclusively on native; web keeps `visibilitychange`.
- (b) Unified `AppLifecycle` service that normalizes both under a single listener.

**Deliverables**:
- Native pause hook wired in `src/db/save-db.ts` or a new `src/lib/app-lifecycle.ts`
- Integration test: simulated Capacitor pause → flush → resume → reload → state restored
- Documented in `docs/ARCHITECTURE.md`

**Owns**: #205 design extended cleanly to native, doesn't regress web.

### Macro M2 — Mobile performance budget + regression harness (P0)

**Why**: Task #22 is open ("Run benchmarks on phone + Mac + live-fire playtest → write decision doc"). Without a real phone benchmark we're shipping blind.

**Branches**:
- (a) CI-run synthetic benchmark on a stable instance (predictable numbers, may not match reality)
- (b) Manual on-device benchmark captured as screenshots/logs, committed to `docs/bench/`
- (c) Expo-style dev build distributed via TestFlight/Firebase App Distribution for beta testers to capture traces

**Deliverables**:
- Phone benchmark numbers on a defined low/mid/high device tier (Pixel 6a / Pixel 9 / iPhone 15)
- Budget doc: p95 frame time, draw calls, triangle count, active InstancedMesh count
- CI regression gate: if `kingdom-gen.bench.test.ts` exceeds prior `.snapshot.json` by > 10%, fail PR
- `playtest.ts` report summarising live-fire run (death rate, session length, chunks visited)

### Macro M3 — Content volume pass (P0)

**Why**: A "shippable beta" with 6 anchor points on a 28km road needs real density. Current state: many pools are stub-sized. If content is thin the pastoral atmosphere reads as empty rather than serene.

**Branches**:
- (a) Author inline (slow but highest quality)
- (b) Generate-then-curate via LLM pass (fast, needs review gate)
- (c) Hybrid: key NPCs and quests inline, ambient chatter via LLM

**Deliverables** (in `content/`):
- ≥ 40 unique NPC blueprints with portraits, dialogue, quest hooks
- ≥ 15 macro quests (1–2h each) covering Ashford → Grailsend beat structure
- ≥ 40 meso quests (15–45min) per-settlement
- ≥ 100 micro quests (5–15min) for wilderness encounters and pilgrim conversations
- ≥ 20 NPC pools with authored variance
- Each settlement (6) has a town config with unique layout + placed buildings + residents

**Gate**: `pnpm validate-content` reports zero schema errors, zero orphan references.

### Macro M4 — Combat feel pass (P0)

**Why**: Shipped #155 gives hit detection. But "hit registers" ≠ "combat feels good". Mobile-first combat needs weight, feedback, and a legible difficulty curve.

**Branches**:
- (a) Action-adventure arcade feel — fast swings, forgiving timing
- (b) Souls-like stamina gating — deliberate, punishing misses
- (c) Hybrid — weapon-class-dependent (daggers fast, cleavers slow)

**Deliverables**:
- Weapon reach + swing arc tuned per archetype (knife vs axe vs cleaver vs machete)
- Hit-pause (frame hitch 60–80ms on successful hit)
- Screen shake on heavy impacts
- Monster stagger state with i-frames
- 3-tier difficulty curve: wild stragglers → dungeon encounters → boss rooms
- Audio pass: hit sfx, whiff sfx, weapon-specific swoosh, monster death cry
- Combat playtest log: average encounter length, player death rate target 15–25%

### Macro M5 — Onboarding + first-hour polish (P0)

**Why**: You have one hour to convince a new player this is worth their time. Currently the first minute is landing → "Begin Pilgrimage" → fall into world with no grounding.

**Branches**:
- (a) Tutorial-free, environmental storytelling (Journey model)
- (b) Minimal overlay (first-time move, first-time attack, first-time pickup shown as diegetic runes)
- (c) Framed prologue — Ashford's innkeeper walks you through the basics

**Deliverables**:
- First-minute flow: intro cinematic → spawn → grounded in Ashford → learn controls → first pilgrim encounter
- First-hour arc: pick up relic → meet first wandering monk → learn about the Grail quest
- Diegetic tutorial that doesn't break the HUD philosophy (no "Press E" prompts — instead, compass pip glows, weapon rune lights up on pickup)
- Main-menu copy that sells the fantasy without exposition dump
- Save-on-first-action so "back tomorrow" always works

### Macro M6 — Mobile UX (touch + haptics) pass (P0)

**Why**: Capacitor-first means touch-first. Current HUD has functional joystick+look+attack zones (#183, #194), but gesture design hasn't been polished.

**Branches**:
- (a) Classic dual-stick (ship as-is, tune thresholds)
- (b) Gesture-first (swipe to attack, hold-to-sprint, two-finger pinch-zoom)
- (c) Hybrid with adaptive hints (joystick fades after confidence, gesture primary)

**Deliverables**:
- Joystick dead-zone + ramp tuned on real devices
- Haptic feedback on: hit landed, gem pickup, quest step, menu tap (via `@capacitor/haptics`)
- Touch-target audit: every interactable ≥ 44pt
- Portrait lock OR landscape-optimal (not both — commit)
- Safe-area insets respected on notched devices
- Thumb reachability: key actions within bottom 1/3 of screen

### Macro M7 — Audio identity pass (P0)

**Why**: Existing ambient mixer (#210-safe) plus combat sfx foundation. Missing: music, wind-over-dialogue ducking, diegetic ambient richness.

**Branches**:
- (a) Generative-ambient (Tone.js) only, no music (matches pastoral mood, avoids licensing)
- (b) Authored music loops per biome/region
- (c) Hybrid — generative ambient, authored cues at key beats

**Deliverables**:
- Biome-specific ambient mixes (Ashford hamlet hum, forest birds, moor wind, dungeon drips)
- Combat music swell on encounter start, fade on victory
- Dialogue ducking (ambient drops to −12dB when DialogueBox opens)
- Grail-theme cue that plays on anchor arrival
- Volume sliders in Settings actually route to the right gain nodes
- Audio mixing doc in `docs/DESIGN.md`

### Macro M8 — Saves + cloud sync (P1)

**Why**: Local save works (post #205). For "1.0 that respects the player", cross-device sync and save exports matter.

**Branches**:
- (a) Ship 1.0 with local-only saves, punt sync to 1.1
- (b) Manual export/import (JSON file download/upload)
- (c) Automated sync via GitHub Gist / iCloud / Google Drive (per-platform)

**Deliverables**:
- Export-save button in pause menu → downloads a JSON file
- Import-save in main menu → upload slot JSON
- For (c): research spike doc on GameCenter iCloud / Play Games Services
- Save format versioning field + migration path

### Macro M9 — Accessibility pass (P1)

**Why**: Not a nice-to-have — excluding players with disabilities is a defect.

**Branches**:
- (a) Minimum viable (subtitles, colourblind safe palette, large-text toggle)
- (b) Full WCAG 2.2 AA pass
- (c) Assistive mode (auto-combat, auto-move, simplified controls)

**Deliverables**:
- Subtitles for all dialogue (currently only text), closed-caption for ambient sfx
- Colourblind simulation audit (protanopia/deuteranopia/tritanopia screenshots)
- Adjustable text size slider in Settings
- Motion-sickness: toggle camera bob, toggle head-bob, toggle chromatic aberration
- Left-handed control scheme (mirror)
- Screen-reader labelling of diegetic interactables (opt-in)
- Documented in `docs/ACCESSIBILITY.md` (new doc)

### Macro M10 — Post-launch content delivery (P1)

**Why**: Content DB is compiled at build-time. New quests = new release. That's fine for beta, awkward for 1.0.

**Branches**:
- (a) Keep compiled-at-build, rely on release cycle
- (b) Hot-reload new content/*.json from CDN on app open
- (c) Remote-first: content DB lives on CDN, app fetches on launch with local cache

**Deliverables**:
- Design doc in `docs/ARCHITECTURE.md` describing choice
- Implementation if (b) or (c) chosen
- Version stamping so old clients don't crash on new schemas

### Macro M11 — Telemetry + crash reporting (P1)

**Why**: Without telemetry we're debugging blind after launch.

**Branches**:
- (a) None (privacy-first, rely on user reports)
- (b) Local-only session logs with opt-in export
- (c) Anonymous crash reporting via Sentry or PostHog (opt-in, documented)

**Deliverables**:
- If (b) or (c): consent dialogue on first launch, privacy policy text, toggle in Settings
- Scrubbing of PII (seed phrases, player names)
- Documented in `docs/DEPLOYMENT.md` and `docs/ACCESSIBILITY.md`

### Macro M12 — Narrative completion (P0)

**Why**: The road spine defines 6 anchor chapters — Grailsend is the endgame. The final beat must land.

**Branches**:
- (a) Canonical ending (single resolution, like pilgrimage reaches chapel)
- (b) Branching endings (3–5 based on choices, moral alignment, quest completion)
- (c) Open-ended (reach Grailsend, game continues in "after the pilgrimage" mode)

**Deliverables**:
- Ending cinematic(s) authored
- Credits screen with production notes, music, asset attributions
- "New Game+" flag if (c) chosen — carries over equipment/XP
- Grailsend settlement layout + final NPC dialogue tree
- Post-ending save state (can continue exploring)

### Macro M13 — Quality gate: real-device beta (P0)

**Why**: We've been deploying to Pages but nobody has played the game end-to-end on a real phone with real thumbs.

**Branches**:
- (a) Internal playtesting only (you + family)
- (b) TestFlight / Play Internal Testing with 5–20 trusted testers
- (c) Open beta on Pages with feedback link

**Deliverables**:
- Feedback-collection mechanism (GitHub issue template, or Discord, or Tally form)
- Minimum 3 playthroughs to Grailsend by someone other than the author
- Bug list triaged and closed to 0 P0/P1 before release

### Macro M14 — Store-ready assets (P0)

**Why**: App stores require screenshots, icons, descriptions. GitHub Pages needs OG tags. Even web-only needs a landing.

**Deliverables**:
- App icon (1024×1024 + adaptive icon for Android)
- Splash screen
- Store screenshots (6.7", 6.5", 5.5" iPhone; 7" + 10" Android tablet)
- OG tags + Twitter card on landing page
- App description (short + long) in `docs/STORE_LISTING.md`
- Privacy policy published to Pages subroute
- Credits and asset attributions

### Macro M15 — Build + release automation hardening (P0)

**Why**: Current release.yml assumes web-only. Native builds are manual.

**Deliverables**:
- `release.yml` builds: web bundle zip, Android debug APK, Android release AAB (unsigned until store), iOS archive (on macOS runner)
- Signing keystore managed via GH secrets (Android upload key)
- SBOM + attestation on all artifacts
- Release candidate tag convention: `v1.0.0-rc.1` → `v1.0.0`
- Automated release notes from changelog

## Acceptance criteria for 1.0

This is the gate the release must pass. If any item is red, we do not tag 1.0.

### Functional

- [ ] New Game → explore Ashford → reach Millbrook (meso anchor) → ≥ 3 quest completions → reach Grailsend (macro anchor) → ending plays → credits → back to main menu
- [ ] Save from anywhere, reload, state restored exactly (position, inventory, quests, time-of-day, weather, dungeon)
- [ ] Force-close mid-play, reopen, Continue lands you within 200m of where you were
- [ ] Full gameplay loop works on: desktop Chrome, desktop Safari, iOS Safari, Android Chrome, Android APK
- [ ] ≥ 15 hours of content in main + side quests

### Performance

- [ ] Pixel 6a (low-tier reference): p95 frame time < 33ms, p99 < 50ms, no drops below 20fps
- [ ] Desktop Chrome: p95 < 16.6ms, p99 < 20ms
- [ ] Initial bundle (gzipped JS) < 2MB
- [ ] First meaningful paint < 2s on 4G connection
- [ ] Memory: no growth > 50MB over 30-minute session (leak regression)

### Quality

- [ ] Zero P0 bugs on issue tracker
- [ ] Zero P1 bugs on issue tracker
- [ ] ≥ 3 external playthroughs completed with reported issues closed
- [ ] Lighthouse score ≥ 80 in all four categories on Pages build
- [ ] Accessibility: WCAG 2.2 AA for UI surfaces
- [ ] Every asset has documented provenance in `docs/ASSETS.md`

### Release-readiness

- [ ] `v1.0.0-rc.1` cut, dogfooded for 7 days, issues triaged, `v1.0.0` cut
- [ ] CHANGELOG complete and reviewed
- [ ] README has embedded gameplay GIF + screenshots
- [ ] Store listing copy finalized
- [ ] Privacy policy published
- [ ] Credits complete

## Execution notes

- This PRD is designed for `/task-batch` autonomous execution. Each macro is a task-group with its own sub-PRD authored at start.
- Branch per macro is the default; some meso items can share a branch if tightly coupled.
- Context compaction survives via `.claude/state/task-batch/`; no handoffs, run until VERIFIED_DONE.
- When a meso or micro completes, its line item in this doc flips from `[ ]` to `[x]` and gets the PR number. Keep this file living.

## Risks + mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Content volume (M3) under-estimated | High | High | Use LLM-pass variant with curator review; track word count weekly |
| Combat feel (M4) requires multiple iteration loops | High | High | Budget 2× the estimate; ship playable-but-rough then polish |
| Real-device perf (M2) blows budget | Med | High | Have a fallback quality tier + dynamic LOD |
| iOS app store review rejects first submission | Med | Med | Ship web + Android first, iOS as 1.0.1 |
| Accessibility pass reveals fundamental HUD assumptions broken | Med | Med | Do the audit (M9) BEFORE shipping diegetic-only; prototype a "bar HUD" fallback toggle |
| Asset license audit finds a proprietary file that can't ship | Low | High | Front-load the audit in MD4; replace offenders early |
| Save migration on format bump breaks existing players | Med | High | Version every save; write migration tests |
| Telemetry opt-in UX regulatory issues (GDPR, CCPA) | Low | Med | Default-off; privacy policy reviewed before collecting anything |
