## Priority: CRITICAL (blocks launch)

## Overview

King's Road is a mobile-first, pastoral-medieval, diegetic-HUD single-player RPG built on Capacitor + Vite + React 19 + React Three Fiber + Koota ECS. This batch covers **every remaining item** to ship a 1.0 Beta release with:

1. **Web build** deployed to GitHub Pages at `https://arcade-cabinet.github.io/kings-road/` — must load, run, and be playable end-to-end.
2. **Debug APK** built in CI, uploaded as PR artifact and release asset — must install on Android and run the same game loop.
3. **Zero procedural appearance code** in the runtime — all meshes, textures, and materials loaded at runtime come only from repo-shipped `public/assets/` (and any explicitly supported remote/CDN sources such as Polyhaven). `pending-integration/` and `/Volumes/home/assets/` are development-only source locations and must be ingested into `public/assets/` before release.
4. **Zustand fully purged** — all state lives in Koota traits + action functions.
5. **Diegetic HUD** — no keyboard overlays, no floating HP bars, no visible stat chrome. Everything in-world.
6. **Capacitor-native persistence** — Preferences for config/settings, `@capacitor-community/sqlite` for save state + content DB.
7. **Feature-complete game loop** — landing → new game → world → combat → quests → save → load → continue.
8. **All CodeRabbit / Gemini / Copilot PR feedback addressed** — no admin-merge shortcuts.
9. **CI/Release/CD workflows green** — dependabot automerged, release-please tagging, APK + web deploys on tag.

## Technical Notes

- **Parallelize aggressively**: Use the Agent tool with background mode for each phase. Phases 3, 7, 8, 9 are independent and can run concurrently.
- **One PR per phase** where practical (per user's "one PR per work stream" rule), with incremental commits triggering CodeRabbit re-review.
- **Never admin-merge** — address all feedback before merge.
- **Capacitor is the primary target**; mobile/touch UX is the default, but this beta also ships a playable web build on GitHub Pages and must be validated end-to-end there. All UX decisions favor touch/mobile.
- **Assets**: check NAS (`/Volumes/home/assets`) before generating anything. Never procedural appearance.
- **Perl not sed** for batch substitutions (macOS sed is unreliable).
- **Biome** for formatting, not Prettier/ESLint.
- **Path aliases**: `@/*` → `src/*`, `@app/*` → `app/*`. Always use aliases, never relative paths across package boundaries.

## Risks

- **Zustand gameStore migration is large (532 LOC)** — likely needs to split into multiple traits. Risk: subtle behavior changes in reactive subscription patterns.
- **Capacitor SQLite on native has platform-specific quirks** — iOS WKWebView and Android WebView may behave differently around file paths and transaction timing.
- **Procgen removal changes visuals significantly** — regression risk if authored assets don't cover all spawned entity types.
- **CI Android job adds ~10min to every PR** — may need to keep it on-demand (workflow_dispatch) rather than on every push.
- **Release-please + automerge interaction** needs careful config to avoid infinite version-bump loops.
- **Legacy save migration** — existing IndexedDB saves must transfer cleanly to SQLite or the upgrade path breaks the player's progress.
- **Asset licensing** — Polyhaven is CC0, PSX packs need license check, custom FBX assets need provenance verified.

- Background agents should be spawned for Phase 3 (procgen kill), Phase 7 (asset sweep), Phase 8 (docs), and Phase 9 (workflows) in parallel with main-thread Phase 1-2 work.
